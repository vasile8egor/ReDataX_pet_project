# M4: прямая регрессия ожидаемой амплитуды убытка

## 1. Назначение

Этот файл описывает прикладную спецификацию модели M4:

- фактический статус реализации;
- место M4 в последовательности M0-M5;
- требования к входным данным;
- построение непрерывной целевой переменной;
- выбор основы признаков;
- базовые и основные алгоритмы;
- обучение и настройку параметров;
- временную проверку;
- калибровку прогноза в базисных пунктах;
- подключение к политике P2;
- прогнозные и экономические метрики;
- структуру сохраняемых артефактов;
- критерии приемки;
- места для ссылок на дашборды.

Математическое обоснование приведено в [THEORY.md](THEORY.md). Общая постановка задачи приведена в [00_problem_statement.md](../../research/00_problem_statement.md).

M4 отвечает на вопрос:

> Можно ли по причинно доступному состоянию рынка напрямую оценить ожидаемую амплитуду будущего неблагоприятного маркаута и использовать эту оценку для экономически оправданного вмешательства?

В отличие от M0-M3, M4 прогнозирует не вероятность события, а величину ожидаемого убытка в базисных пунктах.

## 2. Место в иерархии моделей

| Модель | Выход модели | Основное назначение |
|---|---|---|
| `M0_single_scale` | вероятность неблагоприятного события | одномасштабная точка отсчета |
| `M1_local_multiscale` | вероятность неблагоприятного события | проверка локального многомасштабного описания |
| `M2_cross_market_rg_no_j` | вероятность неблагоприятного события | проверка добавочной информации второго рынка |
| `M3_cross_market_rg_with_j` | вероятность неблагоприятного события | проверка явного межрыночного взаимодействия |
| `M4_direct_value_regression` | ожидаемая амплитуда убытка, б.п. | прямая экономическая оценка для P2 |
| `M5_hurdle_economic_model` | вероятность и условная амплитуда | двухкомпонентная оценка для P3 |

M4 не является автоматическим продолжением M3 по размерности признаков. Она должна использовать последнюю модель признаков, которая прошла критерии приемки на обучающей и проверочной частях.

Допустимые основы:

```text
M0
M1
M2
M3
```

Выбранная основа фиксируется в реестре до открытия итогового теста.

```text
planned
```

## 4. Граница между текущим кодом и целевой спецификацией

| Компонент | Текущий код | Целевая M4 |
|---|---|---|
| тип задачи | бинарная классификация | неотрицательная регрессия |
| единица наблюдения | агрегированная сделка | состояние рынка с шагом 10 секунд |
| стандартные горизонты | 1 и 5 секунд | основной горизонт 600 секунд |
| целевая переменная | `markout_bps > 0` | `max(markout_bps, 0)` |
| выход модели | вероятность | ожидаемый убыток в б.п. |
| набор моделей экономики | M0, M1 | M4 и ориентиры |
| отбор | верхняя доля строк | бюджет по доле номинала |
| порог безубыточности | не применяется | 4 б.п. в конфигурации 1.0 |
| политика | диагностический захват | P2 |
| калибровка величины | отсутствует | обязательна |
| артефакт модели | не сохраняется полностью | полный воспроизводимый пакет |

До реализации регулярной сетки решений допускается диагностический запуск M4 на уровне агрегированных сделок. Такой результат должен иметь отдельный идентификатор:

```text
m4_trade_level_prototype
```

Он не заменяет основной эксперимент версии 1.0.

## 5. Предварительные условия

Перед реализацией M4 должны быть выполнены следующие условия:

- [ ] зафиксирована единая формула знакового маркаута для M0-M5;
- [ ] зафиксирован основной горизонт `600` секунд;
- [ ] зафиксирована длина будущего окна цены или VWAP;
- [ ] реализована сетка решений с шагом `10` секунд;
- [ ] исключено использование будущих данных при построении признаков;
- [ ] реализовано временное разбиение с защитным промежутком;
- [ ] определена последняя принятая основа признаков M0-M3;
- [ ] все варианты M4 используют одну маску наблюдений;
- [ ] определены единицы номинала и базисных пунктов;
- [ ] зафиксированы параметры экономики вмешательства;
- [ ] реализован бюджет по доле номинала, а не по числу строк;
- [ ] итоговый тест не использован для выбора алгоритма, функции потерь и калибровки.

Если условия не выполнены, результат M4 считается диагностическим.

## 9. Единица наблюдения

Целевая версия использует регулярную сетку решений:

```text
t_i = t_0 + i * decision_step_seconds
```

Основная конфигурация:

```text
decision_step_seconds = 10
prediction_horizon_seconds = 600
target_window_seconds = <зафиксировать>
```

Для каждого момента `decision_ts` формируется одна строка:

```text
decision_ts
trade_date
target_symbol
reference_symbol
notional_usdt
feature_backbone
features
markout_bps
adverse_amplitude_bps
target_available_ts
split_name
```

`target_available_ts` нужен для проверки того, что цель обучающего наблюдения уже была доступна к моменту начала следующей части.

## 10. Целевая переменная

Пусть `markout_bps` - знаковый будущий маркаут, где положительное значение соответствует движению против условной экспозиции платформы.

Цель M4:

```text
adverse_amplitude_bps = max(markout_bps, 0.0)
```

Математически:

\[
S_i(H)=\max(M_i(H),0).
\]

Требования:

- цель неотрицательна;
- цель измеряется в базисных пунктах;
- нулевое значение сохраняется как содержательная часть распределения;
- положительные значения не обрезаются без отдельной зарегистрированной абляции;
- одна и та же реализация цели используется в M4, M5, P2, P3 и P4.

Недопустимо:

```text
обучать M4 только на S > 0
```

Такой вариант оценивает условную серьезность и относится ко второй части M5, а не к прямой M4.

## 12. Основа признаков

M4 использует один из ранее проверенных наборов:

```text
m0_single_scale
m1_local_multiscale
m2_cross_market_rg_no_j
m3_cross_market_rg_with_j
```

Рекомендуемый интерфейс:

```text
--feature-backbone m0|m1|m2|m3
```

Основной вариант выбирается на проверочной части по заранее заданному правилу:

1. модель признаков прошла собственные критерии приемки;
2. прирост не объясняется утечкой или несовпадением масок;
3. результат устойчив по дням;
4. сложность оправдана относительно более простой основы;
5. итоговый тест не использовался при выборе.

В карточке запуска сохраняются:

```yaml
feature_backbone: M1
feature_schema_version: local_multiscale_v1
feature_count: 29
```

Пример выше является шаблоном, а не зафиксированным выбором.

## 13. Общая маска наблюдений

Все сравниваемые варианты M4 должны использовать одну маску:

```text
valid_target
AND valid_notional
AND valid_features_for_all_compared_models
AND valid_time_split
AND valid_market_state
```

Если сравниваются M4 с основами M1 и M3, обе модели оцениваются только на пересечении допустимых строк.

В артефактах сохраняются:

```text
rows_before_mask
rows_after_mask
removed_missing_target
removed_missing_features
removed_stale_reference_market
removed_split_embargo
```

Разные числа строк делают прямое сравнение некорректным.

## 14. Простые ориентиры

До обучения основной модели необходимо рассчитать минимум три ориентира.

### 14.1. Нулевой прогноз

```text
prediction_bps = 0.0
```

Проверяет, не достигается ли низкая ошибка только за счет большой доли нулей.

### 14.2. Среднее обучения

```text
prediction_bps = mean(adverse_amplitude_bps on train)
```

Среднее вычисляется только на обучающей части.

### 14.3. Причинное скользящее среднее

Используется только прошлая информация:

```text
prediction_bps(t) = mean(S_j for t - window <= t_j < t)
```

Параметр окна выбирается на проверочной части.

Рекомендуемые идентификаторы:

```text
m4_zero_baseline
m4_train_mean_baseline
m4_causal_rolling_mean_baseline
```

## 15. Основные кандидаты алгоритма

M4 должна содержать не один заранее объявленный алгоритм, а ограниченный зарегистрированный набор кандидатов.

### 15.1. Линейная регрессия с L2-регуляризацией

Назначение:

- простой ориентир;
- проверка линейной добавочной информации;
- интерпретация знаков коэффициентов.

Ограничение:

- возможны отрицательные прогнозы;
- тяжелый хвост сильно влияет на квадратичную потерю.

Идентификатор:

```text
m4_ridge
```

### 15.2. Регрессия Твиди с логарифмической связью

Назначение:

- основная статистическая кандидатура для неотрицательной цели;
- работа с точными нулями и непрерывной положительной частью;
- положительный прогноз без последующего отсечения.

Рекомендуемый идентификатор:

```text
m4_tweedie_glm
```

Настраиваемые параметры:

```text
power
alpha
max_iter
tol
```

Диапазон `power` должен лежать внутри:

```text
1 < power < 2
```

Конкретная сетка фиксируется в конфигурации эксперимента.

### 15.3. Гистограммный градиентный бустинг

Назначение:

- проверка нелинейных зависимостей;
- учет взаимодействий признаков без ручного перечисления всех произведений;
- ограниченное потребление памяти по сравнению с полным хранением деревьев для всех событий.

Рекомендуемые варианты:

```text
m4_hist_gb_squared
m4_hist_gb_poisson
```

Вариант с пуассоновской потерей используется как практический неотрицательный ориентир. Он не должен называться моделью Твиди.

### 15.4. Модель на преобразованной цели

Вариант:

```text
y_log = log1p(adverse_amplitude_bps)
```

Идентификатор:

```text
m4_log_target_regression
```

Он является отдельной абляцией. Обратное преобразование без поправки не оценивает условное среднее исходной цели.

## 16. Рекомендуемая основная конфигурация

Начальная исследовательская конфигурация:


## 17. Ограничения по памяти

При шаге решений 10 секунд число строк существенно меньше числа отдельных сделок. Поэтому целевой набор следует материализовать на уровне решений, а не повторно обучаться на полном потоке `aggTrades`.

Рекомендуемый порядок:

1. прочитать исходные сделки по дням;
2. построить причинные признаки;
3. сформировать строки решений с шагом 10 секунд;
4. рассчитать цель;
5. сохранить компактную таблицу признаков;
6. обучать M4 на этой таблице.

Рекомендуемые форматы:

```text
ClickHouse table
Parquet by symbol/date
```

Не рекомендуется одновременно хранить в памяти:

- полные исходные сделки нескольких месяцев;
- все промежуточные скользящие массивы;
- все варианты признаков M0-M3;
- все матрицы перекрестной проверки.

## 18. Стандартизация

Для линейных моделей параметры стандартизации оцениваются только на обучающей части:

```text
scaler.fit(X_train)
scaler.transform(X_train)
scaler.transform(X_validation)
scaler.transform(X_test)
```

Для последовательного обучения допускается двухпроходная схема:

```text
проход 1: оценка параметров стандартизации
проход 2: обучение модели
```

Нельзя обновлять параметры стандартизации на проверочной или тестовой части.

Для древовидной модели стандартизация обычно не требуется, но набор входных строк и правила пропусков должны совпадать с линейными вариантами.

## 19. Обработка пропусков

Основной вариант:

```text
удалить строку, если отсутствует обязательный причинный признак
```

Заполнение допускается только при выполнении условий:

- правило имеет экономический смысл;
- параметры заполнения рассчитаны на обучении;
- добавлен индикатор пропуска;
- результат сравнен с удалением строк;
- не используется будущее значение.

Для межрыночных основ M2/M3 устаревшее состояние соседнего рынка не заменяется ближайшим будущим наблюдением.

## 20. Обучение

Общая последовательность:

```text
1. Загрузить манифест временных частей.
2. Построить или прочитать общую таблицу решений.
3. Применить единую маску допустимых строк.
4. Рассчитать статистики цели только на train.
5. Обучить простые ориентиры.
6. Обучить кандидаты M4.
7. Настроить параметры только по validation.
8. Зафиксировать выбранную конфигурацию.
9. При необходимости переобучить на train + validation.
10. Один раз рассчитать прогнозы на final test.
11. Откалибровать модель только по заранее разрешенной схеме.
12. Запустить политику P2.
13. Сохранить прогнозные и экономические артефакты.
```

Тестовая часть не используется для:

- выбора основы признаков;
- выбора алгоритма;
- выбора `power`;
- выбора регуляризации;
- выбора преобразования цели;
- выбора порога безубыточности;
- выбора бюджета;
- выбора метода калибровки.

## 21. Временная проверка

Основная схема:

```text

```

Минимальная длина защитного промежутка:

```text
embargo_seconds >= prediction_horizon_seconds + target_window_seconds
```

Рекомендуется использовать расширяющееся окно:

```text
fold 1: train_1 -> validation_1
fold 2: train_1 + train_2 -> validation_2
fold 3: train_1 + train_2 + train_3 -> validation_3
```

Итоговый временной отрезок открывается один раз после фиксации всей спецификации.

Обязательные поля манифеста:

```text

train_end
validation_start
validation_end
test_start
test_end
embargo_seconds
prediction_horizon_seconds
target_window_seconds
```

## 26. Диагностика хвоста

Редкие крупные маркауты могут определять экономическую ценность и одновременно делать обучение нестабильным.

Обязательные срезы:

```text
S = 0
0 < S <= p50_positive
p50_positive < S <= p90
p90 < S <= p99
S > p99
```

Для каждого среза рассчитываются:

```text

mean_prediction_bps
mean_error_bps
rmse_bps
captured_by_P2
```

Крупные значения нельзя удалять только потому, что они ухудшают RMSE.

Допустимые исследования устойчивости:

```text
winsorized_training_only
robust_loss
log_target_ablation
notional_weighted_ablation
```

Все они публикуются отдельно от основной конфигурации.

## 27. Переход к политике P2

M4 передает политике прогноз:

```text
expected_adverse_amplitude_bps
```

Для наблюдения `i` ожидаемая валовая защищенная стоимость:

\[
\widehat G_i
=
N_i\rho_{int}\rho_{mit}
\frac{\widehat S_i}{10^4}.
\]

Стоимость действия:

\[
K_i
=
N_i\frac{c_{act}}{10^4}.
\]

Ожидаемая чистая ценность:

\[
\widehat V_i
=
\frac{N_i}{10^4}
\left(
\rho_{int}\rho_{mit}\widehat S_i-c_{act}
\right).
\]

Основная конфигурация версии 1.0:

```yaml
internalization_rate: 0.25
mitigation_efficiency: 0.50
action_cost_bps: 0.50
notional_budget_fraction: 0.10
break_even_markout_bps: 4.00
```

Безубыточный порог:

```text
4.00 bps = 0.50 / (0.25 * 0.50)
```

Политика рассматривает только наблюдения:

```text
prediction_bps > break_even_markout_bps
```

После этого кандидаты сортируются по убыванию ожидаемой ценности на единицу номинала. При постоянных экономических параметрах это эквивалентно сортировке по `prediction_bps`.

## 28. Алгоритм бюджетного отбора P2

Бюджет задается долей номинала:

```text
sum(selected_notional) / sum(total_notional) <= 0.10
```

Рекомендуемый детерминированный алгоритм:

```text
1. Удалить строки с prediction_bps <= break_even_markout_bps.
2. Отсортировать по prediction_bps по убыванию.
3. При равенстве использовать decision_ts, затем устойчивый row_id.
4. Последовательно добавлять наблюдения.
5. Если добавление следующего наблюдения превышает бюджет, пропустить его.
6. Продолжить просмотр оставшихся строк, если более мелкий номинал помещается в остаток бюджета.
7. Сохранить фактическую долю использованного номинала.
```

Альтернативный вариант с остановкой на первом превышении должен иметь отдельный идентификатор политики.

Отбор верхней доли строк не считается реализацией бюджета по номиналу.

## 29. Реализованный убыток и экономика

Наблюдаемый потенциальный убыток до вмешательства:

\[
L_i=N_i\rho_{int}\frac{S_i}{10^4}.
\]

Реализованная валовая защищенная стоимость:

\[
G_i=a_iN_i\rho_{int}\rho_{mit}\frac{S_i}{10^4}.
\]

Стоимость вмешательства:

\[
K_i=a_iN_i\frac{c_{act}}{10^4}.
\]

Чистая защищенная стоимость:

\[
V=\sum_iG_i-\sum_iK_i.
\]

Для воспроизводимости необходимо хранить отдельно:

```text
observed_loss_usdt
gross_protected_value_usdt
action_cost_usdt
net_protected_value_usdt
```

Обязательный набор:

| Метрика | Определение |
|---|---|
| `selected_notional_fraction` | вовлеченный номинал / общий номинал |

## 32. Отрицательные контроли

Обязательные контроли:

### 32.1. Перемешанная цель

Цель перемешивается внутри обучающей части. Проверочная и тестовая цели не меняются.

Ожидание:

```text
качество близко к постоянному ориентиру
```

### 32.2. Временной сдвиг признаков вперед

Контроль намеренно создает утечку только в отдельном тесте системы контроля.

Ожидание:

```text
leakage_check должен завершить запуск с ошибкой
```

### 32.3. Случайный шумовой блок

К принятой основе добавляются случайные признаки.

Ожидание:

```text
устойчивого улучшения вне выборки нет
```

### 32.4. Постоянный прогноз с тем же средним

Позволяет отделить калибровку общего уровня от полезного ранжирования.

## 33. Контроль утечек

## 48. Типовые ошибки

### Низкая MAE при бесполезной модели

Причина:

```text
большая доля нулей
```

Проверка:

```text
сравнить с нулевым прогнозом и средним обучения
```

### Хорошее ранжирование, но неверный денежный уровень

Причина:

```text
модель различает риск, но систематически завышает или занижает амплитуду
```

Проверка:

```text
калибровочная кривая, mean_bias_bps, slope и intercept
```

### Отличный результат после `log1p`, но слабая экономика

Причина:

```text
обратное преобразование не восстанавливает условное среднее
```

Проверка:

```text
сравнить средний прогноз в исходной шкале и реализованное среднее
```

### P2 нарушает бюджет

Причина:

```text
отбор сделан по доле строк, а не по доле номинала
```

Проверка:

```text
selected_notional_fraction <= B
```

### Порог безубыточности равен 1 б.п.

Причина:

```text
из формулы исключен уровень интернализации, хотя стоимость начисляется на полный номинал
```

Правильная формула основной конфигурации:

```text
0.50 / (0.25 * 0.50) = 4.00 bps
```

### M4 лучше по RMSE, но хуже по P2

Причина:

```text
ошибки сосредоточены около порога или на крупных номиналах
```

Проверка:

```text
ошибка по квантилям прогноза, номинала и около 4 б.п.
```

### Результат определяется одним днем

Проверка:

```text
дневные разности, leave-one-day-out и повторная выборка по дням
```

### M4 обучена только на положительных целях

Следствие:

```text
это модель условной серьезности, а не прямая M4
```

Такой компонент должен быть перенесен в M5.

## 49. Известные ограничения и технический долг

На текущем этапе:

- M4 не реализована;
- основной горизонт 600 секунд не перенесен в действующий контур;
- сетка решений 10 секунд не реализована в текущем эксперименте;
- полные M2 и M3 еще не реализованы;
- основа признаков M4 не выбрана;
- единый контракт цели требует фиксации;
- функция потерь не выбрана экспериментально;
- калибровка величины не реализована;
- политика P2 не реализована;
- бюджет текущего скрипта задается по числу наблюдений;
- полный пакет артефактов моделей не сохраняется;
- дашборды M4 отсутствуют;
- сценарные параметры экономики пока постоянны;
- не моделируются динамические комиссии, ликвидность и влияние на рынок;
- публичные `aggTrades` не содержат реального клиентского потока платформы;
- денежные результаты не являются фактической прибылью банка или биржи.
