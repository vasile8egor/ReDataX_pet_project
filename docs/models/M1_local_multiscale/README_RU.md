# M1: локальная многомасштабная модель потока

## 1. Назначение

Этот файл описывает практическую реализацию модели `M1`: источник данных, порядок построения многомасштабных признаков, обучение, временную проверку, запуск экспериментов, структуру выходных файлов и правила сравнения с M0.

Математическое обоснование приведено в [THEORY.md](THEORY.md), общая постановка задачи - в [00_problem_statement.md](../../research/00_problem_statement.md).

M1 проверяет гипотезу:

> Состояние направленного потока одного рынка, измеренное одновременно на нескольких масштабах, содержит дополнительную информацию о будущем неблагоприятном движении цены по сравнению с одним фиксированным масштабом.

M1 не является самостоятельной точкой отсчета. Ее результат интерпретируется только в парном сравнении с M0 на тех же наблюдениях, датах, горизонтах и целевой переменной.

## 2. Место в иерархии моделей

| Модель | Информационное множество |
|---|---|
| `M0_single_scale` | один рынок, один масштаб |
| `M1_local_multiscale` | один рынок, несколько масштабов |
| `M2_cross_market_rg_no_j` | связанные рынки без явного попарного взаимодействия |
| `M3_cross_market_rg_with_j` | связанные рынки с явным попарным взаимодействием |
| `M4_direct_value_regression` | прямая оценка ожидаемой амплитуды убытка |
| `M5_hurdle_economic_model` | вероятность события и условная амплитуда убытка |

Переход M0 -> M1 изменяет только набор признаков. Алгоритм обучения, целевая переменная и временные части должны оставаться одинаковыми.

## 3. Где реализована M1

| Контур | Файл | Назначение |
|---|---|---|
| первичная проверка | `src/revolut_app/real_market/experiments/adverse_selection_mvp.py` | ранняя проверка моделей на одном торговом периоде |

## 6. Единица наблюдения и причинность

В текущей реализации одна строка выборки соответствует агрегированной сделке с индексом `i`.

Перед расчетом данных выполняется сортировка:

```text
(event_timestamp, aggregate_trade_id)
```

Историческое окно для сделки `i` включает только сделки с меньшими индексами. Текущая сделка не входит в расчет дисбаланса, а будущие сделки используются только при построении целевой переменной.

Для масштаба `k` берутся ровно `k` предшествующих событий. Поэтому масштаб отражает не физическое время, а глубину истории в количестве сделок. В период высокой активности одно окно может покрывать доли секунды, а в период низкой активности - существенно больший промежуток.

## 7. Расчет многомасштабного поля

Для каждой предшествующей сделки определяется подписанный номинал:

```text
signed_notional = aggressor_sign * notional_usdt
```

На масштабе `k` рассчитывается нормированный дисбаланс:

```text
phi_k = sum(signed_notional на последних k сделках)
        / sum(notional_usdt на последних k сделках)
```

Используемая сетка:

```python
SCALES = (1, 2, 4, 8, 16, 32, 64)
```

Для каждого наблюдения получается профиль:

```text
phi_1, phi_2, phi_4, phi_8, phi_16, phi_32, phi_64
```

Все семь значений должны быть конечными. Поэтому первые 64 сделки каждого торгового дня исключаются из M0, M1 и ранней M2 общей маской. Это обеспечивает парное сравнение моделей на одинаковых строках.

## 8. Целевая переменная

Текущая реализация использует знаковый маркаут:

```text
markout_bps = aggressor_sign
              * (future_vwap - current_price)
              / current_price
              * 10_000
```

`future_vwap` рассчитывается по сделкам, попавшим в окно:

```text
[t_i + horizon_seconds,
 t_i + horizon_seconds + target_window_seconds]
```

Бинарная цель:

```text
y = 1, если markout_bps > 0
y = 0, если markout_bps <= 0
```

M1 оценивает вероятность положительного знакового маркаута. Изменение определения будущей цены, горизонта или знака цели требует новой версии набора данных и повторного обучения всех сравниваемых моделей.

Каноническое определение должно храниться в [targets.md](../../ml/targets.md).

## 9. Признаки M1

M1 использует один признак размера наблюдения и четыре преобразования поля на каждом из семи масштабов.

### 9.1. Логарифм номинала

```text
log_notional = log(1 + notional_usdt)
```

Признак описывает размер текущей сделки и ограничивает влияние редких крупных значений.

### 9.2. Согласованный поток

```text
aligned_k = aggressor_sign_i * phi_k
```

Положительное значение означает, что предшествующий поток совпадает с направлением текущей сделки.

### 9.3. Амплитуда дисбаланса

```text
absolute_k = abs(phi_k)
```

Этот признак описывает силу направленного потока независимо от его знака.

### 9.4. Нелинейные преобразования

```text
q2_k = phi_k ** 2
q4_k = phi_k ** 4
```

Преобразования позволяют линейному классификатору различать слабые и сильные дисбалансы. Они не интерпретируются как физический потенциал без дополнительного обоснования.

### 9.5. Порядок столбцов

Матрица признаков формируется в следующем порядке:

```text
1. log_notional
2. aligned_1 ... aligned_64
3. absolute_1 ... absolute_64
4. q2_1 ... q2_64
5. q4_1 ... q4_64
```

Итоговая размерность:

```text
1 + 7 + 7 + 7 + 7 = 29 признаков
```

Рекомендуемые канонические имена:

```text
log_notional
aligned_s1, aligned_s2, aligned_s4, aligned_s8,
aligned_s16, aligned_s32, aligned_s64
abs_phi_s1, abs_phi_s2, abs_phi_s4, abs_phi_s8,
abs_phi_s16, abs_phi_s32, abs_phi_s64
phi2_s1, phi2_s2, phi2_s4, phi2_s8,
phi2_s16, phi2_s32, phi2_s64
phi4_s1, phi4_s2, phi4_s4, phi4_s8,
phi4_s16, phi4_s32, phi4_s64
```

Схема и порядок столбцов должны быть закреплены в [feature_engineering.md](../../ml/feature_engineering.md) и сохраняться вместе с артефактом модели.

## 10. Отличие M1 от M0

M0 использует те же типы преобразований, но только для масштаба 16:

```text
log_notional,
aligned_16,
absolute_16,
q2_16,
q4_16
```

M1 использует полный профиль на семи масштабах. Таким образом:

```text
M0: 5 признаков
M1: 29 признаков
```

Для корректного вывода M0 и M1 должны обучаться:

- на одних и тех же датах;
- на одной целевой переменной;
- на одинаковой маске наблюдений;
- одним алгоритмом;
- с одинаковой силой регуляризации;
- без выбора параметров по итоговому тесту.

В текущем коде эти требования в основном выполняются внутри одного запуска `adverse_selection_oos.py`.

## 11. Алгоритм обучения

Для каждой пары `рынок x горизонт` создается отдельная модель:

```text
StandardScaler
SGDClassifier(
    loss="log_loss",
    penalty="l2",
    alpha=1e-4,
    learning_rate="optimal",
    average=True,
    random_state=20260627,
)
```

Обучение выполняется в два прохода по обучающим дням:

1. `StandardScaler.partial_fit()` оценивает средние и стандартные отклонения признаков.
2. `SGDClassifier.partial_fit()` обучает логистическую модель на стандартизированных признаках.

Матрицы признаков хранятся как `float32`, а данные обрабатываются по одному торговому дню. Это снижает пиковое потребление памяти.

### 11.1. Что пока не реализовано

- подбор коэффициента регуляризации `alpha`;
- выбор или исключение масштабов по проверочной части;
- сравнение разных сеток масштабов;
- отдельная калибровка вероятностей;
- сохранение модели и стандартизатора;
- сохранение явного списка признаков;
- повторное обучение на объединении обучения и проверки после фиксации конфигурации;
- автоматическая регистрация результата в `MODEL_REGISTRY.md`.

Проверочная часть сейчас используется для независимого измерения, но не для систематического выбора конфигурации.

## 12. Временная проверка

В `adverse_selection_oos.py` задаются три непересекающихся календарных периода:

```text
train_start ... train_end
validation_start ... validation_end
test_start ... test_end
```

Рекомендуемый порядок:

```text
обучение -> проверка -> итоговый тест
```

Код проверяет отсутствие пересечения дат, но не формирует отдельный защитный интервал между частями. Для основной постановки с горизонтом 600 секунд необходимо исключать наблюдения, целевое окно которых пересекает следующую часть.

Единицей оценки неопределенности является торговый день. Повторная выборка отдельных строк не используется, поскольку соседние наблюдения имеют пересекающиеся исторические и целевые окна.

Полный протокол должен быть закреплен в [temporal_validation.md](../../ml/temporal_validation.md).

## 13. Прогнозные метрики

Для каждого дня и объединенного периода сохраняются:

| Поле | Смысл |
|---|---|
| `observations` | число допустимых наблюдений |
| `toxic_rate` | доля положительного класса |
| `roc_auc` | качество попарного ранжирования классов |
| `average_precision` | качество ранжирования положительного класса |
| `brier_score` | средняя квадратичная ошибка вероятностного прогноза |
| `top_decile_lift` | отношение доли положительного класса в верхнем дециле прогноза к общей доле |

Основная исследовательская величина:

```text
M1 average_precision - M0 average_precision
```

Дополнительно рассчитываются парные дневные изменения:

```text
M1 - M0 ROC-AUC
M1 - M0 Average Precision
M0 - M1 Brier score
M1 - M0 top-decile lift
```

Для каждого изменения сохраняются:

- среднее по дням;
- 2.5-й и 97.5-й процентили повторной выборки дней;
- доля дней с положительным изменением.

Average Precision должна интерпретироваться вместе с долей положительного класса. Значение метрики без базовой частоты события недостаточно для оценки качества.

## 14. Экономический контур

`adverse_selection_capture.py` обучает M0 и M1, ранжирует наблюдения по прогнозной вероятности и измеряет, какая часть исторически наблюдаемого неблагоприятного маркаута попала в верхнюю долю списка.

Текущая оценка убытка:

```text
adverse_loss_usdt = notional_usdt
                    * max(markout_bps, 0)
                    / 10_000
```

Она не учитывает сценарные параметры `rho_int`, `rho_mit` и стоимость действия `c_act`. Поэтому это оценка наблюдаемой неблагоприятной экспозиции, а не чистая защищенная стоимость из полной постановки задачи.

Стандартные доли отбора:

```text
1%, 5%, 10%, 20%
```

Отбор выполняется отдельно внутри каждого тестового дня по точному числу наблюдений с наибольшим прогнозом.

### 14.1. Метрики захвата

| Поле | Смысл |
|---|---|
| `selected_trade_fraction` | доля выбранных наблюдений |
| `selected_notional_fraction` | доля номинала выбранных наблюдений |
| `captured_adverse_loss_usdt` | сумма наблюдаемого убытка в выбранной группе |
| `capture_rate` | доля общего наблюдаемого убытка, попавшая в выбранную группу |
| `loss_lift` | отношение среднего убытка выбранных наблюдений к общему среднему |
| `capture_lift_vs_random` | превосходство над случайным отбором той же доли наблюдений |
| `captured_loss_per_million_total_notional` | захваченный убыток на 1 млн USDT общего номинала |
| `oracle_capture_rate` | результат идеального ранжирования по известному будущему убытку |
| `oracle_efficiency` | доля результата идеального ранжирования, достигнутая моделью |

Для M1 сохраняется парная разность относительно M0 по каждому тестовому дню и доле отбора.

### 14.2. Ограничение текущего контура

Текущая политика ограничивает долю числа наблюдений, а не долю вовлеченного номинала:

```text
selected_observations / observations
```

В постановке версии 1.0 бюджет задается как:

```text
selected_notional / total_notional <= B
```

До финального экономического отчета отбор должен быть переведен на точное ограничение по номиналу.

## 16. Проверка наличия данных

Перед запуском следует проверить покрытие рынков по датам:

```bash
docker compose exec clickhouse clickhouse-client \
  --user default \
  --password default \
  --query "
    SELECT
        trade_date,
        symbol,
        count() AS rows,
        min(event_timestamp) AS first_event,
        max(event_timestamp) AS last_event
    FROM raw.fact_real_market_agg_trades FINAL
```

Эксперимент не следует запускать, если:

- отсутствует часть требуемых дат;
- рынок содержит меньше 65 событий за день;
- временные метки нарушают порядок;
- на оцениваемом дне отсутствует один из классов;
- целевое будущее окно не заполнено для значительной части наблюдений.

## 17. Запуск вневыборочного эксперимента

Ниже приведен шаблон команды. Даты следует заменить на фактически загруженные периоды.

```bash
mkdir -p artifacts/real_market

docker compose exec airflow-scheduler \
  python -m revolut_app.real_market.experiments.adverse_selection_oos \
  --symbols BTCUSDT ETHUSDT \
  --horizons-seconds 1 5 \
  --target-window-seconds 1 \
  --train-start 2026-01-01 \
  --train-end 2026-01-14 \
  --validation-start 2026-01-15 \
  --validation-end 2026-01-21 \
  --test-start 2026-01-22 \
```

После завершения в журнал выводятся средняя дневная разность AP и доверительный интервал для:

```text

M2 - M1
```

## 19. Структура результата OOS

Файл `adverse_selection_oos.json` имеет структуру:

```text
configuration
symbols
  <symbol>
    validation
      <horizon>
        daily
        pooled
        bootstrap
    test
      <horizon>
        daily
        pooled
        bootstrap
```

Основные поля M1:

```text
symbols.<symbol>.test.<horizon>.daily[*].models.m1_multiscale
symbols.<symbol>.test.<horizon>.pooled.m1_multiscale
symbols.<symbol>.test.<horizon>.bootstrap.m1_minus_m0
```

Пример логики итогового чтения:

```text
pooled.m1_multiscale.average_precision
bootstrap.m1_minus_m0.average_precision.mean
bootstrap.m1_minus_m0.average_precision.ci_025
bootstrap.m1_minus_m0.average_precision.ci_975
bootstrap.m1_minus_m0.average_precision.positive_day_fraction
```

## 20. Структура результата capture

Файл `adverse_selection_capture.json` имеет структуру:

```text
configuration
symbols
  <symbol>
    <horizon>
      daily
        <date>
          capacities
            <fraction>
              models
                m0_single_scale
                m1_multiscale
              m1_minus_m0
      aggregate
        <fraction>
          m0_single_scale
          m1_multiscale
      bootstrap
        <fraction>
          captured_adverse_loss_usdt
          capture_rate
          loss_lift
          captured_loss_per_million_total_notional
          oracle_efficiency
```

Для дашбордов следует использовать объединенные показатели вместе с распределением дневных разностей. Только объединенный результат может быть обусловлен одним крупным или аномальным днем.

## 21. Проверки после запуска

- [ ] обучающий, проверочный и итоговый периоды не пересекаются;
- [ ] рынки и горизонты совпадают у M0 и M1;
- [ ] обе модели используют одинаковые допустимые наблюдения;
- [ ] первые 64 события каждого дня исключены общей маской;
- [ ] в каждом оцениваемом блоке присутствуют оба класса;
- [ ] число признаков M1 равно 29;
- [ ] порядок признаков соответствует разделу 9.5;
- [ ] стандартизатор обучен только на обучающем периоде;
- [ ] итоговый тест не использовался для выбора масштабов или регуляризации;
- [ ] AP сравнивается с долей положительного класса;
- [ ] показатель Брайера не интерпретируется без калибровочной диаграммы;
- [ ] доверительные интервалы рассчитаны по торговым дням;
- [ ] экономические показатели не называются фактической прибылью;
- [ ] ограничение по числу наблюдений не подменяет бюджет по номиналу;
- [ ] JSON сохранен в каталоге артефактов;
- [ ] версия кода и даты данных внесены в реестр моделей;
- [ ] вывод сформулирован отдельно для BTCUSDT и ETHUSDT;
- [ ] дашборды обновлены ссылками на конкретный запуск.

## 22. Критерии приемки M1

M1 считается подтвержденным расширением M0, если одновременно выполняются следующие условия:

1. средняя дневная разность AP на итоговом периоде положительна;
2. доверительный интервал не указывает на устойчивое ухудшение;
3. улучшение наблюдается не только в объединенной выборке, но и в значимой доле дней;
4. показатель Брайера не ухудшается критически;
5. при одинаковой доле отбора M1 захватывает не меньшую долю неблагоприятной экспозиции, чем M0;
6. результат не определяется одним рынком, горизонтом или аномальным днем без явного объяснения;
7. вывод сохраняется при разумном изменении регуляризации и состава масштабов на проверочной части;
8. все признаки проходят контроль причинности и утечки.

Статус `accepted` не означает готовность к производственному развертыванию. Он означает, что многомасштабные признаки дали воспроизводимую добавочную ценность относительно M0 в рамках эксперимента.

## 26. Известные ограничения и технический долг

1. Масштабы измеряются числом сделок, а не временем.
2. Решение принимается на каждой сделке, а не с шагом 10 секунд.
3. Стандартные горизонты 1 и 5 секунд не соответствуют основному горизонту 600 секунд.
4. Соседние масштабы сильно коррелированы из-за вложенных окон.
5. Состав масштабов и регуляризация зафиксированы в коде.
6. Вклад отдельных масштабов и групп масштабов пока не оценивается автоматически.
7. Калибровка вероятностей отсутствует.
8. Модель и стандартизатор не сохраняются как повторно используемые артефакты.
9. Явный договор о порядке 29 признаков не проверяется тестом.
10. Защитный интервал между временными частями отсутствует.
11. Экономический контур ограничивает долю наблюдений, а не долю номинала.
12. Контур захвата не учитывает интернализацию, эффективность смягчения и стоимость действия.
13. M1 не использует данные соседнего рынка и не описывает межрыночную передачу сигнала.
14. Бинарная цель не различает малые и крупные неблагоприятные движения.
15. Публичные `aggTrades` не содержат клиентский тип, инвентарь платформы, книгу заявок и реальные затраты исполнения.
16. Использование нескольких масштабов само по себе не является доказательством ренормгруппового поведения.

Устранение пунктов 1-12 требует новой версии реализации и повторной фиксации результатов M0 и M1.

## 29. Переход к M2

После фиксации результата M1 следующий эксперимент должен ответить на вопрос:

> Содержит ли состояние связанного рынка дополнительную информацию после учета полного локального многомасштабного профиля?

Для этого M2 должна сохранить локальные признаки M1 и добавить межрыночное состояние без явного попарного члена взаимодействия. Сравнение M2 - M1 должно проводиться по тем же правилам, что и сравнение M1 - M0.
