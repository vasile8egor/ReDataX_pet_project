# M2: межрыночная многомасштабная модель без явного взаимодействия

## 1. Назначение

Этот файл описывает практическую спецификацию модели M2: ее место в последовательности экспериментов, текущий прототип в коде, целевую межрыночную реализацию, порядок построения признаков, синхронизацию рынков, обучение, временную проверку, выходные артефакты и критерии приемки.

Математическое обоснование приведено в [THEORY.md](THEORY.md), общая постановка задачи - в [00_problem_statement.md](../../research/00_problem_statement.md).

M2 должна ответить на вопрос:

> Содержит ли состояние связанного рынка дополнительную информацию о будущем неблагоприятном движении целевого рынка после учета его собственного многомасштабного состояния?

Целевая M2 использует два рынка, но не вводит явные произведения их полей. Поэтому модель оценивает аддитивный вклад локального и соседнего рынков. Явное межрыночное взаимодействие проверяется только в M3.

## 2. Место в иерархии моделей

| Модель | Информационное множество | Основное сравнение |
|---|---|---|
| `M0_single_scale` | один рынок, один масштаб | точка отсчета |
| `M1_local_multiscale` | один рынок, несколько масштабов | M1 против M0 |
| `M2_cross_market_rg_no_j` | два рынка, несколько масштабов, без явного взаимодействия | M2 против M1 |
| `M3_cross_market_rg_with_j` | два рынка с явными межрыночными произведениями | M3 против M2 |
| `M4_direct_value_regression` | прямая оценка ожидаемой амплитуды убытка | экономическое ранжирование |
| `M5_hurdle_economic_model` | вероятность события и условная амплитуда | двухкомпонентная оценка риска |

Переход M1 -> M2 должен изменять только информационное множество. Целевая переменная, алгоритм обучения, временные части, маска наблюдений и процедура оценки должны оставаться одинаковыми.

## 3. Ключевое различие между спецификацией и кодом

В репозитории уже существует модель с именем:

```text
m2_rg_flow
```

Однако эта реализация пока не является полной `M2_cross_market_rg_no_j`.

Текущий `m2_rg_flow`:

- использует только один рынок за запуск;
- наследует 29 локальных признаков M1;
- добавляет 18 локальных признаков изменения по масштабу;
- не присоединяет состояние второго рынка;
- не проверяет направления `BTCUSDT <- ETHUSDT` и `ETHUSDT <- BTCUSDT`;
- не может подтвердить межрыночную гипотезу.

Поэтому в документации используются два отдельных обозначения:

| Обозначение | Смысл |
|---|---|
| `m2_rg_flow` | текущий локальный прототип размерности 47 |
| `m2_cross_market_rg_no_j` | целевая межрыночная модель размерности 93 |

До миграции рекомендуется зарегистрировать текущую модель как `M1_local_rg_flow_v0` или сохранить статус `diagnostic_only`.

## 7. Направления межрыночного эксперимента

M2 оценивается отдельно в двух направлениях:

```text
BTCUSDT <- ETHUSDT
ETHUSDT <- BTCUSDT
```

Левая часть является целевым рынком A, правая - связанным рынком B.

Направления не следует объединять в одну итоговую цифру без отдельного отчета, поскольку передача информации может быть несимметричной.

Рекомендуемые идентификаторы запусков:

```text
m2_btcusdt_from_ethusdt
m2_ethusdt_from_btcusdt
```

## 8. Единица наблюдения

### 8.1. Текущий прототип

В `adverse_selection_oos.py` одна строка соответствует отдельной агрегированной сделке целевого рынка. Все признаки строятся по предыдущим событиям того же рынка.

### 8.2. Целевая реализация

В версии 1.0 решение принимается на регулярной сетке:

```text
t_i = t_0 + i * 10 секунд
```

Для каждого момента `t_i` строятся:

- признаки целевого рынка A;
- признаки связанного рынка B;
- целевая переменная только для рынка A;
- метаданные свежести присоединенного состояния B.

Одному моменту решения должна соответствовать ровно одна строка признакового набора.

## 9. Причинная синхронизация рынков

Для каждого момента решения `t_i` используются только события с временной меткой:

```text
event_timestamp <= t_i
```

Состояние рынка B присоединяется назад по времени. Ближайшая будущая сделка запрещена.

Рекомендуемый алгоритм:

1. построить регулярную сетку решений для рынка A;
2. независимо рассчитать оконные признаки A и B;
3. выполнить `as-of join` с направлением `backward`;
4. сохранить время последнего события B;
5. вычислить возраст связанного состояния;
6. применить заранее установленное правило устаревания;
7. сформировать общую маску M1 и M2.

Обязательные поля синхронизации:

```text
decision_ts
last_event_ts_a
last_event_ts_b
staleness_ms_a
staleness_ms_b
is_stale_b
```

## 10. Правило устаревания состояния B

Пусть:

```text
staleness_b = decision_ts - last_event_ts_b
```

Порог `max_staleness_ms` должен выбираться только по обучающей и проверочной частям.

Допустимые стратегии:

| Стратегия | Описание |
|---|---|
| удаление | строка исключается, если B слишком старый |
| индикатор | значение сохраняется вместе с `is_stale_b` |
| причинное заполнение | используется последнее допустимое значение в пределах порога |

Основная стратегия должна быть одинаковой для всех сравниваемых конфигураций M2 и фиксироваться в метаданных запуска.

Если M2 удаляет наблюдение из-за рынка B, M1 необходимо пересчитать на той же маске.

## 12. Разность согласованного поля

В коде:

```python
delta_aligned = np.diff(aligned, axis=1)
```

Для соседних масштабов:

```text
delta_aligned_s1_s2
 delta_aligned_s2_s4
 delta_aligned_s4_s8
 delta_aligned_s8_s16
 delta_aligned_s16_s32
 delta_aligned_s32_s64
```

Определение:

```text
delta_aligned_k_2k = aligned_2k - aligned_k
```

Число признаков:

```text
6
```

Положительное значение означает, что согласованное направление сильнее на более длинном масштабе. Отрицательное значение означает, что короткий импульс сильнее длинного фона.

## 18. Что означает отсутствие явного взаимодействия

M2 не содержит признаков вида:

```text
phi_a_s16 * phi_b_s16
aligned_a_s8 * aligned_b_s8
phi_a_sm * phi_b_sn
```

Классификатор может независимо использовать состояние A и B, но не получает специально сконструированный признак совместного режима.

Это ограничение является намеренным. Сначала необходимо установить, дает ли сам соседний рынок добавочную информацию. Только после этого M3 проверяет явные межрыночные взаимодействия.

## 19. Целевая переменная

Текущий код использует знаковый маркаут:

```text
markout_bps = aggressor_sign
              * (future_vwap - current_price)
              / current_price
              * 10_000
```

Бинарная цель:

```text
y = 1, если markout_bps > 0
y = 0, если markout_bps <= 0
```

В целевой версии:

- маркаут рассчитывается только для рынка A;
- связанный рынок B не участвует в определении цели;
- основной горизонт равен 600 секундам;
- решение принимается с шагом 10 секунд;
- все определения должны совпадать с [targets.md](../../ml/targets.md).

Будущее рынка B запрещено использовать как признак.

## 20. Алгоритм обучения

Текущий прототип использует тот же алгоритм, что M0 и M1:

```text
StandardScaler
SGDClassifier(
    loss="log_loss",
    penalty="l2",
    alpha=1e-4,
    learning_rate="optimal",
    average=True,
    random_state=20260627,
)
```

Обучение выполняется в два прохода по обучающим дням:

1. оценка параметров стандартизации через `partial_fit`;
2. обучение логистической модели через `partial_fit`.

Для корректного сравнения M1 и M2 должны использовать:

- один и тот же алгоритм;
- одинаковую силу регуляризации либо общий протокол ее выбора;
- одинаковые даты;
- одну целевую переменную;
- одну маску допустимых строк;
- одинаковую процедуру калибровки.

Для 93 признаков рекомендуется проверить сетку `alpha` на проверочной части, поскольку фиксированное значение `1e-4` может по-разному влиять на M1 и M2.

## 21. Временная проверка

Используется последовательное разбиение:

```text
train -> validation -> test
```

Текущий скрипт проверяет только отсутствие пересечения календарных дат. В целевой реализации дополнительно требуется защитный интервал не меньше горизонта прогнозирования.

Для горизонта 600 секунд:

```text
purge_gap >= 600 секунд
```

Наблюдение удаляется, если его будущее целевое окно пересекает границу следующей части.

По проверочной части выбираются:

- коэффициент регуляризации;
- порог устаревания B;
- правило пропусков;
- состав блоков признаков;
- границы ограничения логарифмических изменений;
- допустимая задержка B;
- способ калибровки.

Итоговый тест не используется для выбора этих параметров.

## 22. Основные сравнения

### 22.1. Текущий прототип

В `adverse_selection_oos.py` автоматически рассчитывается:

```text
m2_rg_flow - m1_multiscale
```

Это сравнение проверяет только добавочную ценность локальных изменений по масштабу.

### 22.2. Полная M2

Основное сравнение:

```text
m2_cross_market_rg_no_j - m1_multiscale
```

Оно проверяет совместный вклад:

- связанного рынка;
- локальных изменений по масштабу;
- изменений по масштабу связанного рынка.

Для содержательной интерпретации обязательны разложения:

| Конфигурация | Проверяемый вклад |
|---|---|
| M1 | базовое локальное состояние |
| M1 + `A_scale_flow` | локальные изменения по масштабу |
| M1 + `B_state` | состояние соседнего рынка |
| M1 + `B_state` + `B_scale_flow` | полный вклад B без локальных производных A |
| полная M2 | все четыре блока |

## 23. Отрицательные контроли

Полная M2 не принимается без отрицательных контролей.

### 23.1. Перемешивание дней рынка B

Дни рынка B случайно переставляются относительно A, сохраняя внутреннюю структуру каждого дня.

Ожидание:

```text
прирост M2 относительно M1 должен исчезнуть или существенно уменьшиться
```

### 23.2. Причинный сдвиг рынка B

Признаки B дополнительно задерживаются на заранее заданные интервалы:

```text
1 с, 5 с, 10 с, 30 с, 60 с
```

Это показывает чувствительность к свежести межрыночного сигнала.

### 23.3. Случайный шумовой блок

Вместо B добавляется блок случайных признаков той же размерности.

Если шумовой блок дает сопоставимое улучшение, результат может быть связан с процедурой обучения или нестабильной оценкой.

### 23.4. Общая маска

Все контроли, M1 и M2 оцениваются на одинаковых строках.

## 24. Контроль утечек

## 33. Экономическая проверка

- концентрации риска;
- валовой защищенной стоимости;
- стоимости вмешательств;
- чистой защищенной стоимости;
- коэффициенту выгоды-затрат;
- эффективности относительно P4.

Текущий `adverse_selection_capture.py` не оценивает M2 и ограничивает долю числа наблюдений. Поэтому для M2 требуется отдельное расширение экономического контура.

## 34. Критерии приемки M2

M2 получает статус `accepted`, если одновременно выполняются следующие условия:

1. полная реализация действительно использует состояние рынка B;
2. M1 и M2 оцениваются на общей маске;
3. средняя дневная разность AP M2 - M1 положительна на итоговом периоде;
4. доверительный интервал не указывает на устойчивое ухудшение;
5. улучшение не определяется одним торговым днем;
6. перемешивание дней B устраняет или существенно ослабляет эффект;
7. случайный шумовой блок не воспроизводит улучшение;
8. эффект сохраняется при небольшой причинной задержке B;
9. калибровка не ухудшается критически;
10. при одинаковом бюджете M2 захватывает не меньшую экспозицию, чем M1;
11. результат воспроизводится хотя бы для одного направления A <- B;
12. все проверки временной причинности проходят автоматически.

Статус текущего `m2_rg_flow` может быть только:

```text
diagnostic_only
```

даже при положительной разности AP, поскольку он не содержит межрыночных признаков.

## 39. Известные ограничения и технический долг

1. Текущий `m2_rg_flow` не использует второй рынок.
2. Масштабы текущего кода измеряются числом сделок, а не временем.
3. Решение принимается на каждой сделке, а не с шагом 10 секунд.
4. Стандартные горизонты 1 и 5 секунд не соответствуют основному горизонту 600 секунд.
5. Отдельный межрыночный набор данных не реализован.
6. Причинное присоединение рынков отсутствует.
7. Порог устаревания связанного рынка не определен.
8. Отрицательные контроли отсутствуют.
9. Разложение вклада блоков признаков не автоматизировано.
10. Регуляризация фиксирована и не учитывает рост размерности.
11. Калибровка вероятностей отсутствует.
12. Модель, стандартизатор и схема признаков не сохраняются.
13. Экономический контур пока не включает M2.
14. Бюджет текущего capture-контура задается долей наблюдений, а не номинала.
15. Защитный временной интервал отсутствует.
16. Публичные сделки не содержат книгу заявок, клиентский тип, инвентарь платформы и реальные издержки.
17. Наблюдаемая межрыночная связь не доказывает причинность.
18. Признаки изменения по масштабу являются инженерным приближением, а не строгой ренормализационной группой.

## 43. Переход к M3

Переход к M3 допустим только после того, как полная M2 реализована и отдельно подтверждено, что состояние рынка B дает добавочную информацию относительно M1.

M3 должна отвечать на более узкий вопрос:

> Меняется ли влияние состояния одного рынка в зависимости от состояния другого рынка?

Для этого к аддитивному информационному множеству M2 добавляются явные межрыночные произведения и параметры взаимодействия. Если M2 не превосходит M1, сложность M3 должна быть обоснована отдельно, а не вводиться автоматически.
