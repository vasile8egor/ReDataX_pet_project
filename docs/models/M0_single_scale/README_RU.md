# M0: одномасштабная модель локального потока

## 1. Назначение

Этот файл описывает практическую реализацию модели `M0`: источник данных, единицу наблюдения, признаки, обучение, временную проверку, запуск экспериментов и структуру результатов.

Математическое обоснование приведено в [THEORY.md](THEORY.md), общая постановка задачи - в [00_problem_statement.md](../../research/00_problem_statement.md).

M0 проверяет базовую гипотезу:

> Состояние направленного потока одного рынка на одном масштабе содержит информацию о будущем неблагоприятном движении цены.

M0 является точкой отсчета для M1-M3. Она не рассматривается как самостоятельная производственная модель.

## 2. Место в иерархии моделей

| Модель | Информационное множество |
|---|---|
| `M0_single_scale` | один рынок, один масштаб |
| `M1_local_multiscale` | один рынок, несколько масштабов |
| `M2_cross_market_rg_no_j` | несколько связанных рынков без явных попарных произведений |
| `M3_cross_market_rg_with_j` | несколько рынков и явные попарные взаимодействия |
| `M4_direct_value_regression` | прямая оценка ожидаемой амплитуды убытка |
| `M5_hurdle_economic_model` | вероятность события и условная амплитуда убытка |

При сравнении M0-M3 должны совпадать исходные наблюдения, цель, временные периоды, алгоритм обучения и метрики. Меняется только набор признаков.

## 3. Где реализована M0

| Контур | Файл | Назначение |
|---|---|---|
| первичная проверка | `src/revolut_app/real_market/experiments/adverse_selection_mvp.py` | один торговый день, хронологическое разбиение 60/20/20 |
| вневыборочная проверка | `src/revolut_app/real_market/experiments/adverse_selection_oos.py` | обучение, проверка и тест по отдельным календарным периодам |
| оценка захвата убытка | `src/revolut_app/real_market/experiments/adverse_selection_capture.py` | оценка ранжирования при ограниченной доле вмешательств |

Основным контуром статистического сравнения является `adverse_selection_oos.py`. Экономический контур запускается после подтверждения прогнозного сигнала.

## 4. Текущая реализация и целевая спецификация

Документация версии 1.0 и текущий экспериментальный код пока не полностью совпадают.

| Параметр | Реализовано сейчас | Целевая спецификация |
|---|---|---|
| единица наблюдения | отдельная `aggTrade` | состояние рынка с шагом 10 секунд |
| масштаб M0 | фиксированные 16 предшествующих сделок | временное окно, выбранное на проверочном периоде |
| стандартные горизонты | 1 и 5 секунд | 600 секунд |
| будущая цена | VWAP короткого будущего окна | определение из `docs/ml/targets.md` |
| настройка модели | параметры зафиксированы в коде | настройка только по обучению и проверке |
| калибровка | не выполняется | отдельный этап при необходимости |
| артефакт модели | не сохраняется | версионированная модель и параметры преобразования |

Поэтому текущий код следует считать экспериментальным предшественником целевой M0. Результаты разных постановок нельзя объединять в одну таблицу без явного указания версии.

## 5. Источник данных

Основной вневыборочный эксперимент читает данные из ClickHouse:

```text
raw.fact_real_market_agg_trades
```

Используются поля:

| Поле | Назначение |
|---|---|
| `event_timestamp` | время сделки |
| `aggregate_trade_id` | вторичный ключ сортировки |
| `price` | цена сделки |
| `base_quantity` | объем в базовом активе |
| `quote_quantity` | объем в котируемом активе |
| `aggressor_side` | сторона агрессора |
| `trade_date` | календарная дата |
| `symbol` | торговая пара |

Для `BTCUSDT` и `ETHUSDT` номинал равен `quote_quantity`. Для `ETHBTC` номинал переводится в USDT по последней доступной цене `BTCUSDT`.

## 6. Единица наблюдения и причинность

В текущем коде строка выборки соответствует агрегированной сделке с индексом `i`.

Признаки сделки `i` используют только сделки с меньшими индексами. Текущая сделка не включается в окно истории. Перед расчетом данные сортируются по:

```text
(event_timestamp, aggregate_trade_id)
```

Если временные метки не упорядочены, эксперимент завершается с ошибкой.

## 7. Целевая переменная

Текущая реализация рассчитывает знаковый маркаут:

```text
markout_bps = aggressor_sign
              * (future_vwap - current_price)
              / current_price
              * 10_000
```

Здесь `future_vwap` рассчитывается на окне, которое начинается через `horizon_seconds` после текущей сделки и имеет длину `target_window_seconds`.

Бинарная цель:

```text
y = 1, если markout_bps > 0
y = 0, если markout_bps <= 0
```

Положительная метка означает, что цена изменилась в направлении, неблагоприятном для условной пассивной стороны сделки.

Каноническое определение цели должно храниться в [targets.md](../../ml/targets.md). Изменение знаковой конвенции требует новой версии набора данных и повторного запуска всех моделей.

## 8. Признаки M0

Пусть `phi_16` - нормированный дисбаланс подписанного номинала на последних 16 предшествующих событиях, а `s_i` - знак текущей сделки.

Основная вневыборочная реализация использует пять признаков:

| № | Признак | Реализация | Интерпретация |
|---:|---|---|---|
| 1 | логарифм номинала | `log1p(notional_usdt)` | размер текущей сделки |
| 2 | согласованный поток | `s_i * phi_16` | совпадает ли предыдущий поток с направлением текущей сделки |
| 3 | модуль поля | `abs(phi_16)` | сила дисбаланса независимо от знака |
| 4 | квадрат поля | `phi_16 ** 2` | нелинейная зависимость от амплитуды |
| 5 | четвертая степень | `phi_16 ** 4` | повышенный вес сильных дисбалансов |

В `adverse_selection_mvp.py` использовалась ранняя версия из трех признаков: логарифм номинала, согласованный поток и модуль поля. Поэтому результаты MVP и OOS относятся к разным версиям M0.

### 8.1. Особенность общей маски наблюдений

M0 использует только масштаб 16, однако допустимая строка определяется после проверки полей на всех масштабах `(1, 2, 4, 8, 16, 32, 64)`.

Из выборки исключаются первые 64 события каждого дня. Это сделано для сравнения M0, M1 и ранней M2 на одинаковых наблюдениях. Изменение этого правила требует нового идентификатора эксперимента.

## 9. Алгоритм

Используется последовательность:

```text
StandardScaler
SGDClassifier(
    loss="log_loss",
    penalty="l2",
    alpha=1e-4,
    learning_rate="optimal",
    average=True,
    random_state=20260627,
)
```

Обучение проходит в два этапа:

1. `StandardScaler.partial_fit()` оценивает средние и стандартные отклонения по обучающим дням.
2. `SGDClassifier.partial_fit()` оценивает параметры логистической модели после стандартизации.

Данные обрабатываются по одному дню, а матрицы признаков хранятся как `float32`. Это снижает потребление памяти при локальном запуске.

### 9.1. Пока не реализовано

- подбор `alpha` по проверочному периоду;
- выбор масштаба M0;
- калибровка вероятностей;
- повторное обучение после выбора конфигурации;
- сериализация модели и стандартизатора;
- автоматическая регистрация результата в `MODEL_REGISTRY.md`.

Проверочный период сейчас используется для независимой оценки, а не для настройки гиперпараметров.

## 10. Временная проверка

Для `adverse_selection_oos.py` интервалы задаются явно:

```text
train_start ... train_end
validation_start ... validation_end
test_start ... test_end
```

Они не должны пересекаться. Рекомендуемый порядок:

```text
обучение -> проверка -> итоговый тест
```

Каждый торговый день рассматривается как отдельный блок. Доверительные интервалы строятся повторной выборкой торговых дней, а не отдельных сделок.

Текущий код проверяет отсутствие пересечения календарных диапазонов, но не создает дополнительный защитный интервал между ними. Для горизонтов, способных пересечь границу дня или периода, этот контроль должен быть добавлен в [temporal_validation.md](../../ml/temporal_validation.md).

## 11. Метрики

Основная метрика M0 - `average_precision`.

Дополнительно сохраняются:

| Поле JSON | Смысл |
|---|---|
| `observations` | число наблюдений |
| `toxic_rate` | доля положительного класса |
| `roc_auc` | качество попарного ранжирования классов |
| `average_precision` | качество ранжирования положительных событий |
| `brier_score` | квадратичная ошибка вероятностного прогноза |
| `top_decile_lift` | отношение доли токсичных событий в верхнем дециле к общей доле |

Для сравнения M1 с M0 рассчитываются разности метрик по каждому дню и доверительные интервалы для среднего дневного эффекта.

## 15. Запуск оценки захвата неблагоприятной экспозиции

```bash
docker compose exec airflow-scheduler \
  python -m revolut_app.real_market.experiments.adverse_selection_capture \
  --symbols BTCUSDT ETHUSDT \
  --horizons-seconds 1 5 \
  --target-window-seconds 1 \
  --capacity-fractions 0.01 0.05 0.10 0.20 \
  --train-start 2026-01-01 \
  --train-end 2026-01-21 \
  --test-start 2026-01-22 \
  --test-end 2026-01-28 \
  --bootstrap-samples 5000 \
  --output /opt/airflow/artifacts/real_market/adverse_selection_capture.json
```

В этом контуре наблюдения сортируются по прогнозу M0. Для каждой доли пропускной способности выбирается верхняя часть списка внутри каждого тестового дня.

Текущая реализация использует ограничение по числу наблюдений, а не точное ограничение по доле номинала из постановки задачи. Это различие должно быть устранено до финального экономического отчета.

## 16. Структура результата OOS

Файл `adverse_selection_oos.json` имеет следующую логическую структуру:

```text
configuration
symbols
  BTCUSDT | ETHUSDT
    validation | test
      horizon
        daily
        pooled
        bootstrap
```

`daily` содержит метрики каждого торгового дня. `pooled` рассчитывается на объединении всех наблюдений периода. `bootstrap` содержит средний дневной эффект и 95-процентный интервал для сравнений:

```text
m1_minus_m0
m2_minus_m1
```

Для M0 основными полями являются:

```text
symbols.<symbol>.test.<horizon>.daily[*].models.m0_single_scale
symbols.<symbol>.test.<horizon>.pooled.m0_single_scale
```

## 17. Структура результата capture

Файл `adverse_selection_capture.json` содержит результаты по рынку, горизонту и доле отобранных наблюдений.

Основные показатели M0:

| Поле | Смысл |
|---|---|
| `selected_trade_fraction` | фактическая доля выбранных наблюдений |
| `selected_notional_fraction` | доля номинала выбранных наблюдений |
| `captured_adverse_loss_usdt` | наблюдаемый убыток, попавший в выбранную группу |
| `capture_rate` | доля всего наблюдаемого неблагоприятного убытка |
| `loss_lift` | отношение среднего убытка выбранных наблюдений к общему среднему |
| `capture_lift_vs_random` | превосходство над случайным выбором той же доли наблюдений |
| `captured_loss_per_million_total_notional` | захваченный убыток на 1 млн USDT общего номинала |
| `oracle_efficiency` | доля результата идеального ранжирования по известному будущему убытку |

Эти значения описывают наблюдаемую историческую экспозицию. Они не являются реальной прибылью банка или биржи.

## 18. Проверки после запуска

- [ ] обучающий, проверочный и тестовый периоды не пересекаются;
- [ ] для каждого дня присутствуют все требуемые рынки;
- [ ] число наблюдений ненулевое;
- [ ] в каждом оцениваемом блоке присутствуют оба класса;
- [ ] `average_precision` сравнивается с долей положительного класса;
- [ ] `brier_score` не интерпретируется без проверки калибровки;
- [ ] доверительный интервал рассчитан по дням;
- [ ] итоговый тест не использовался для выбора конфигурации;
- [ ] JSON сохранен в каталоге артефактов;
- [ ] версия кода и даты данных внесены в реестр моделей;
- [ ] соответствующие карточки и дашборды обновлены;
- [ ] вывод сформулирован отдельно для BTCUSDT и ETHUSDT.

## 19. Критерии приемки M0

M0 считается корректно реализованной базовой моделью, если:

1. признаки построены причинно;
2. итоговый период не участвовал в настройке;
3. модель воспроизводится по одной команде;
4. результат содержит AP, ROC-AUC, Brier score и подъем верхнего дециля;
5. доступны дневные и объединенные метрики;
6. неопределенность оценена по торговым дням;
7. M1 сравнивается с M0 на одинаковых строках;
8. экономический анализ явно отделен от прогнозного;
9. все расхождения между текущим кодом и целевой спецификацией зарегистрированы.

## 22. Известные ограничения и технический долг

1. Масштаб `16` означает число сделок, хотя теоретическая спецификация оперирует временем.
2. Решение принимается на каждой сделке, а не с шагом 10 секунд.
3. Горизонты 1 и 5 секунд не соответствуют основному горизонту 600 секунд.
4. Масштаб и сила регуляризации зафиксированы в коде.
5. Отдельная калибровка вероятностей отсутствует.
6. Модель не сохраняется как повторно используемый артефакт.
7. Проверочный период не используется для выбора конфигурации.
8. Экономический контур ограничивает число наблюдений, а не долю номинала.
9. Публичные `aggTrades` не содержат клиентский тип, инвентарь платформы и реальные затраты хеджирования.
10. Бинарная цель не различает малые и крупные неблагоприятные движения.

Устранение пунктов 1-8 должно сопровождаться новой версией M0 и повторной фиксацией результатов.
